# الإعدادات العامة عند التثبيت

> **الموضوع:** هيكل **معالج الإعداد الأولي** — ما يحدده المستخدم مرة واحدة عند تشغيل البرنامج.  
> **قرار:** مجموعة حقول **مقفولة** بعد أول عملية تشغيلية — لا إعادة فتح إلا بترحيل استثنائي (لاحقاً).  
> **آخر تحديث:** 2026-07-05 · **الحالة:** 🟡 موثّق — مقترح هيكل معتمد للتخطيط

---

## 1) لماذا ملف واحد للإعدادات العامة؟

الإعدادات موزّعة اليوم بين [ملف](README.md)، [المواد](../المواد%20والمستودعات/README.md)، [الحسابات](../الحسابات/README.md)، [الفواتير](../الفواتير/تعريف-الفواتير.md) — لكن **المستخدم يراها خطوة واحدة** عند التثبيت.

| المبدأ | التطبيق |
|--------|---------|
| **مصدر حقيقة واحد** | هذا الملف + جدول `company_settings` (أو `system_foundation_settings`) |
| **معالج متسلسل** | خطوات واضحة — لا تشتيت بين شاشات |
| **قفل صريح** | ما يُقفل يُعلَم للمستخدم **قبل** التأكيد |
| **تفاصيل per قسم** | تبقى في وثائق الأقسام — هنا **الفهرس والترتيب** فقط |

---

## 2) ما يُقفل vs ما يبقى مفتوحاً

### 2.1) مقفول بعد أول فاتورة/حركة مخزنية/قيد مرحّل

| الإعداد | المفتاح المقترح | الوثيقة التفصيلية |
|---------|-----------------|-------------------|
| طريقة الجرد | `inventory_method` | [إعدادات المخزون §2](../المواد%20والمستودعات/إعدادات-المخزون-العامة.md) |
| نظام التكلفة | `costing_method` | [إعدادات التكلفة](../المواد%20والمستودعات/إعدادات-التكلفة.md) |
| فصل تكلفة per مستودع | `cost_per_warehouse` | [§5](../المواد%20والمستودعات/إعدادات-المخزون-العامة.md) |
| فصل تكلفة per مركز كلف | `cost_per_cost_center` | [§5](../المواد%20والمستودعات/إعدادات-المخزون-العامة.md) |
| تتبع مخزون مع كل حركة | `track_quantity_on_movement` | دائماً `true` — [قرار #21](../قرارات-المنتج.md) |

### 2.2) قابل للتعديل لاحقاً (أمثلة)

| الإعداد | السبب |
|---------|--------|
| بيانات الشركة، الشعار | هوية — لا يمس التكلفة التاريخية |
| ترقيم السندات/الفواتير per نمط | تشغيلي |
| حسابات افتراضية في الأنماط | تشغيلي |
| صلاحيات المستخدمين | حوكمة |
| فروع ومستودعات **جديدة** | توسع — ضمن نفس قواعد التكلفة المقفولة |

## 3) معالج التثبيت — الخطوات المقترحة

```
┌─────────────────────────────────────────────────────────────┐
│  معالج الإعداد الأولي                                        │
├─────────────────────────────────────────────────────────────┤
│ ① هوية المنشأة                                              │
│ ② المحاسبة والجرد        ← قفل                            │
│ ③ التكلفة والمخزون       ← قفل                            │
│ ④ الفروع والمستودعات     (اختياري — يُنصح به)              │
│ ⑤ الافتتاح               (قيد افتتاحي + بضاعة أول مدة)     │
│ ⑥ مراجعة وتأكيد           «لن تستطيع تغيير ②③ لاحقاً»      │
└─────────────────────────────────────────────────────────────┘
```

### الخطوة ① — هوية المنشأة

| الحقل | إلزامي | ملاحظة |
|--------|--------|--------|
| اسم الشركة | ✅ | |
| العملة الأساسية | ✅ | [الحسابات](../الحسابات/README.md) |
| بداية السنة المالية | ✅ | فترات محاسبية |
| الدولة / المنطقة الزمنية | موصى | تقريب عملة، تقارير |

### الخطوة ② — المحاسبة والجرد ⚠️ قفل

| السؤال | الخيارات |
|--------|----------|
| طريقة الجرد | **دوري** · **مستمر** |
| ماذا يعني؟ | يحدد **قالب قيود المبيعات والمشتريات** — انظر [§2](../المواد%20والمستودعات/إعدادات-المخزون-العامة.md) |

> عرض توضيحي قصير: «مستمر = تكلفة البيع مع كل فاتورة · دوري = تكلفة البيع آخر الفترة».

### الخطوة ③ — التكلفة والمخزون ⚠️ قفل

| السؤال | الخيارات |
|--------|----------|
| نظام التكلفة | متوسط مرجح · FIFO · معياري · … |
| تكلفة مختلفة بين المستودعات؟ | نعم / لا |
| تكلفة مختلفة بين مراكز الكلف؟ | نعم / لا |

**تأثير المناقلة + التكلفة الإضافية:** [§4](../المواد%20والمستودعات/إعدادات-المخزون-العامة.md) — في **الدوري والمستمر**.

### الخطوة ④ — الفروع والمستودعات

| البند | القرار |
|--------|--------|
| إلزامية | موصى بها قبل أول مناقلة أو إغلاق حركات متقدم |
| الحد الأدنى | فرع واحد + مستودع واحد على الأقل إن وُجد مخزون |

### الخطوة ⑤ — الافتتاح

| البند | الربط |
|--------|--------|
| قيد افتتاحي حسابات | [قيد افتتاحي](../سندات/قيد-افتتاحي.md) |
| بضاعة أول مدة | [تعريف الفواتير §1.5](../الفواتير/تعريف-الفواتير.md) |

### الخطوة ⑥ — مراجعة وتأكيد

- ملخص **كل** الخيارات المقفولة
- Checkbox: «أفهم أن تغيير الجرد/التكلفة يتطلب نظاماً جديداً»
- عند التأكيد: `foundation_locked_at = now()`

## 4) نموذج بيانات موحّد (مقترح)

```text
company_foundation_settings
  -- ① هوية
  company_name, base_currency_id, fiscal_year_start, timezone

  -- ② جرد (مقفول)
  inventory_method: periodic | perpetual

  -- ③ تكلفة (مقفول)
  costing_method: weighted_avg | fifo | standard | ...
  cost_per_warehouse: boolean
  cost_per_cost_center: boolean
  track_quantity_on_movement: true   -- ثابت

  -- قفل
  foundation_locked: boolean
  foundation_locked_at timestamptz
  first_posted_transaction_at timestamptz nullable

-- وثائق تفصيلية تبقى في:
--   المواد/إعدادات-المخزون-العامة.md
--   المواد/إعدادات-التكلفة.md
```

## 5) كيف تتصل الأقسام بالإعداد العام

| القسم | ما يقرأ من الإعداد العام |
|--------|---------------------------|
| [الفواتير](../الفواتير/تعريف-الفواتير.md) | قالب القيد من `inventory_method`؛ تكلفة المناقلة من §5 |
| [المواد](../المواد%20والمستودعات/README.md) | كل حركة مخزنية |
| [الحسابات](../الحسابات/README.md) | حسابات مخزون/تكلفة/مشتريات |
| [سندات](../سندات/README.md) | قيد افتتاحي |
| [تقارير](../تقارير/README.md) | جرد دوري → تسوية آخر فترة |

**قاعدة تنفيذ:** لا يقرأ نمط فاتورة أو سند «جرد مستمر» محلياً — يستدعي `resolveFoundationSettings(company_id)`.

## 6) رأي التصميم (لماذا هذا الهيكل)

| البديل | المشكلة | الحل المعتمد |
|--------|---------|--------------|
| إعدادات متفرقة في كل قسم | المستخدم يضيع؛ خطر تناقض | **معالج واحد** + وثيقة فهرس |
| السماح بتغيير الجرد لاحقاً | كسر تاريخ القيود والتكلفة | **قفل** مع تحذير صريح |
| تكلفة المناقلة للدوري فقط | ناقص — المستمر يحتاجها أيضاً | **كلا النظامين** — اختلاف **القيد** فقط |
| كل شيء في `company_settings` مسطح | صعب الصيانة | مجموعة `foundation_*` أو JSONB مُوثَّق |
