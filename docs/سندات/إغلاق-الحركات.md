# إغلاق الحركات — المواصفة المستهدفة

> **قرار معتمد:** تطوير كبير يغطي المقاصة بين مراكز الكلف، الإغلاق من الطرفين (مدين/دائن)، الفروع، وقيود التسوية بين CC والفروع.  
> **الأولوية:** عالية — يتطلب دقة عالية لكل حالات المستخدم.  
> **آخر تحديث:** 2026-07-04 (مقاصة+نقد، استحقاق، سيناريو ز) · **الحالة:** 📋 موثّق — لم يُنفَّذ

---

## 1) الهدف الواجب الوصول إليه

سند **إغلاق حركات** متقدم يستطيع:

1. **جلب الحركات غير المغلقة** — سواء كانت **مدينة** أو **دائنة** على الحساب/الطرف
2. **تخصيص** مبالغ الإغلاق على حركات محددة مع رصيد مفتوح **حقيقي** (أصلي − مُغلق سابقاً)
3. **المقاصة** بين **مراكز الكلف** (وتوسّع لاحقاً بين **فروع**)
4. **نشر/توزيع** السند حسب سياسة مراكز الكلف والفروع (نمط السند يحدد السلوك)
5. عند الحاجة: **توليد قيود تسوية** بين مراكز كلف أو فروع — ليس فقط قيد قبض/دفع تقليدي

**النتيجة المرجوة:** المستخدم يغلق ذمماً وتسويات CC/فرع في مستند واحد محكم، دون أخطاء رصيد أو ازدواجية إغلاق.

## 4) مفاهيم أساسية

### 4.1) الحركة المفتوحة (Open Item)

كل **سطر قيد مرحّل** (`journal_entry_lines` حيث `journal_entries.status = posted`) يمكن أن يكون «مفتوحاً» جزئياً أو كلياً:

```
original_amount = debit أو credit (الطرف غير الصفري)
allocated_amount = مجموع voucher_allocations على هذا السطر (مرحّلة)
open_amount = original_amount − allocated_amount
open_side = debit | credit  (من أي طرف نشأ الرصيد)
```

**قاعدة:** لا يُسمح بتخصيص `applied_amount > open_amount`.

### 4.2) الإغلاق من الطرفين

| نوع الرصيد المفتوح | معنى محاسبي | مثال |
|---------------------|-------------|------|
| **مدين مفتوح** | علينا / مطلوب تحصيل / ذمم مدينة | فاتورة بيع آجلة |
| **دائن مفتوح** | لنا / مطلوب سداد / ذمم دائنة | فاتورة شراء آجلة |

السند يجب أن يعرض **الاثنين** عند الحاجة ويسمح بإغلاق كل نوع ضمن نفس العملية أو حسب نمط السند.

### 4.3) المقاصة بين مراكز الكلف (Netting)

**المقاصة:** تسوية أرصدة مفتوحة بين CC₁ وCC₂ (أو أكثر) — و**قد تتضمن نقداً** ضمن نفس سند الإغلاق.

#### سياسة النقد في المقاصة (قرار معتمد)

| البند | القرار |
|--------|--------|
| هل المقاصة تشمل نقداً؟ | **نعم** — ليست مقاصة محاسبية فقط |
| الافتراضي | يُحدَّد **لكل مركز كلف** في بطاقة CC: `netting_includes_cash_default` |
| التعديل عند الإدخال | مستخدم بصلاحية `vouchers.closure.override_netting` يغيّر لكل سند عند الحاجة |
| النمط | نمط السند قد يفرض أو يقترح الافتراضي من CC |

**أمثلة:**

```
أ) مقاصة محاسبية فقط: CC-أ مدين 10,000 ↔ CC-ب دائن 10,000 → قيد تسوية CC
ب) مقاصة + نقد: دفعة 80,000 نقداً + توزيع على CC-أ وCC-ب مع مقاصة جزئية بينهما
```

يتطلب:
- تجميع حركات مفتوحة **حسب `cost_center_id`**
- قواعد: أي CC يُقاصى مع أي CC (نمط السند / إعدادات / مجموعات CC)
- قيود ناتجة: **تسوية CC** + **قبض/دفع** للجزء النقدي
- حسابات تسوية بين مراكز (وسيط أو per CC — يُحدَّد لاحقاً)

### 4.4) الاستحقاق وأيام السداد (Payment Terms)

يُطبَّق عند ترتيب التغطية والأهلية للسداد:

| المستوى | الحقل | الاستخدام |
|---------|--------|-----------|
| **العميل** | `payment_terms_days` | افتراضي لكل فواتير العميل (مثلاً 30 يوماً) |
| **الفاتورة** | `due_date` أو `payment_terms_days` | يتفق عليه per فاتورة — يُلغي أو يعدّل افتراض العميل |
| **السطر** | من الفاتورة/المرتجع | CC وفرع على مستوى السطر |

**تاريخ الاستحقاق:**

```
due_date = invoice_date + payment_terms_days
(أو due_date صريح على الفاتورة)
```

**في سند إغلاق الحركات:**

- عرض الحركات مع: تاريخ الاستحقاق، متأخر/ضمن المدة، **مؤهل للتسديد** أم لا
- سياسة التوزيع (قابلة للإعداد في النمط):
  - `due_first` — الأقدم استحقاقاً أو المتأخر أولاً
  - `manual` — المستخدم يحدد الفواتير/الأسطر
  - `agreed_basis` — حسب اتفاق مسجّل (عميل/فاتورة)
- **تغطية جزئية:** الدفعة لا تغطي كل المفتوح — يُوزَّع على ما يختاره المستخدم أو ما تسمح به سياسة الاستحقاق

**علاقة:** [عميل](../عميل/README.md) · [الفواتير](../الفواتير/README.md)

### 4.5) الفروع

نفس منطق CC لكن بعد إضافة `branches`:

- فلترة الحركات المفتوحة per فرع
- مقاصة بين فروع
- قيود تسوية بين فروع عند اللزوم
- نشر السند: «يُطبَّق على فرع X فقط» أو «عبر فروع محددة»

**تبعية:** [ملف](../ملف/README.md) — جدول `branches` + `branch_id` على القيود والسندات.

### 4.6) الحسابات الوسيطة (قرار معتمد)

حسابات التسوية (بين CC، بين فروع، أو حساب مقابل) تُعرَّف **افتراضياً** ويمكن **تعديلها في السند**:

| المستوى | أين يُحدَّد الافتراضي | التعديل في السند |
|---------|------------------------|-------------------|
| نمط السند | `default_inter_cc_account_id`, `default_counter_account_id`… | حقول قابلة للتعديل في النموذج |
| مركز الكلف | حساب تسوية افتراضي per CC (اختياري) | يُورَّث ويُعدَّل |
| إعدادات شركة | حسابات وسيطة عامة | — |

**قواعد:**

- الافتراضي يظهر تلقائياً عند فتح السند
- التعديل في السند: لكل مستخدم بصلاحية مناسبة (مثلاً `vouchers.edit` أو `vouchers.closure.override_accounts`)
- عند الترحيل: يُستخدم الحساب **الظاهر في السند** وليس المخزَّن سابقاً في النمط فقط

**أمثلة حسابات وسيطة:**

- تسوية بين مراكز الكلف
- تسوية بين فروع
- حساب مقابل (صندوق/بنك) في إغلاق الحركات

## 5) أنماط السند (إغلاق حركات) — خيارات التعريف

يُعرَّف كـ **نمط سند** ضمن [أنماط غير محدودة](تعريف-أنماط-السندات.md):

| خيار النمط | الوصف |
|------------|--------|
| `closure_mode` | `party` (عميل/مورد) · `account` · `cost_center` · `branch` · `mixed` |
| `allow_debit_open` | جلب وإغلاق مدين مفتوح |
| `allow_credit_open` | جلب وإغلاق دائن مفتوح |
| `allow_netting_cc` | تفعيل مقاصة مراكز كلف |
| `netting_includes_cash_default` | افتراضي النمط (يُورَّث من CC إن وُجد) |
| `allow_override_netting_cash` | السماح بتجاوز الافتراضي بصلاحية |
| `allocation_policy` | `due_first` · `manual` · `agreed_basis` |
| `allow_partial_coverage` | تغطية جزئية للدفعة |
| `allow_netting_branch` | تفعيل مقاصة فروع |
| `allowed_cc_ids` / `cc_group` | حصر CC المسموحة |
| `allowed_branch_ids` | حصر الفروع |
| `auto_inter_cc_entry` | توليد قيد تسوية CC تلقائياً |
| `auto_inter_branch_entry` | توليد قيد تسوية فرع |
| `default_counter_account_id` | حساب مقابل (صندوق/بنك/تسوية) — افتراضي |
| `default_inter_cc_account_id` | حساب وسيط تسوية CC — افتراضي |
| `default_inter_branch_account_id` | حساب وسيط تسوية فرع — افتراضي |
| `allow_override_accounts` | السماح بتغيير الحسابات في السند |
| `require_full_allocation` | إجبار أن يطابق مجموع التخصيصات مبلغ السند |

## 6) سيناريوهات المستخدم (يجب دعمها)

### سيناريو أ — سداد ذمم عميل (مدين مفتوح)

1. اختيار عميل → جلب فواتير/قيود **مدينة مفتوحة**
2. إدخال مبلغ التحصيل → توزيع على حركات (يدوي أو تسلسلي)
3. ترحيل → قيد قبض + تقليل `open_amount`

### سيناريو ب — سداد لمورد (دائن مفتوح)

1. اختيار مورد → جلب **دائن مفتوح**
2. تخصيص → قيد دفع

### سيناريو ج — مقاصة بين مركزي كلف (مع أو بدون نقد)

1. اختيار CC مصدر وCC هدف (أو قائمة حركات متعددة)
2. النظام يعرض أرصدة مفتوحة لكل CC
3. المستخدم يحدد مبالغ المقاصة؛ الجزء النقدي حسب افتراض CC (قابل للتعديل بصلاحية)
4. ترحيل → قيد تسوية CC + قيد قبض/دفع للنقد إن وُجد

### سيناريو ز — مثال معتمد: عميل، فواتير متعددة، مرتجعات، CC مختلفة، دفعة جزئية

**الوضع:**

- عميل «س» لديه عدة **فواتير بيع**؛ أسطر كل فاتورة على **مراكز كلف مختلفة** (CC-1، CC-2، CC-3…)
- **مرتجعات جزئية** على بعض الفواتير؛ أسطر المرتجع على CC **قد تختلف** عن أسطر الفاتورة الأصلية
- العميل أرسل **دفعة** (تحويل/نقد) وطلب **تغطية جزء** من ذمته — وليس بالضرورة كل الفواتير
- **أيام الاستحقاق:** 30 يوماً على بطاقة العميل، أو 60 يوماً على فاتورة معيّنة حسب الاتفاق

**الحركات المفتوحة الناتجة (مفاهيمياً):**

| المصدر | الطرف | CC | مبلغ مفتوح | استحقاق |
|--------|--------|-----|------------|---------|
| فاتورة #101 سطر 1 | مدين | CC-1 | 40,000 | 2026-08-01 |
| فاتورة #101 سطر 2 | مدين | CC-2 | 25,000 | 2026-08-01 |
| فاتورة #102 | مدين | CC-3 | 50,000 | 2026-09-15 |
| مرتجع #R-01 | دائن | CC-1 | 15,000 | — |
| مرتجع #R-02 | دائن | CC-2 | 5,000 | — |

**صافٍ تقريبي per CC (بعد المرتجعات):**

- CC-1: مدين 25,000
- CC-2: مدين 20,000
- CC-3: مدين 50,000
- **إجمالي ذمم العميل:** 95,000

**الدفعة الواردة:** 60,000 — التغطية الجزئية المطلوبة

**ما يجب أن يفعله سند إغلاق الحركات:**

1. جلب كل المفتوح **مدين ودائن** للعميل مع CC لكل سطر
2. عرض الاستحقاق: أي فواتير **مستحقة** وأيها **لم يحن موعدها** (حسب عميل/فاتورة)
3. المستخدم (أو سياسة `due_first`) يحدد:
   - تسديد فاتورة #101 بالكامل بعد خصم المرتجعات ذات الصلة
   - جزء من #102 (مثلاً 35,000) والباقي يبقى مفتوحاً
4. **توزيع الـ 60,000** على أسطر/قيود محددة — ليس بالضرورة ترصيداً أعمى على كل الفواتير
5. **مقاصة:** إن وُجدت حركة دائنة (مرتجع) على CC-1 والتغطية على CC-1 — تُقاصى ضمن نفس CC قبل أو مع النقد
6. **نقد:** الباقي بعد المقاصة يُغلق بقيد قبض على الحساب البنكي
7. **قيود تسوية CC** إن أُغلق CC-1 من دفعة تُنسَب عملياً لـ CC-2 (حسب سياسة النشر) — حسب افتراض المقاصة + النقد per CC
8. بعد الترحيل: تحديث `open_amount` لكل سطر؛ الباقي 35,000+ على #102 يظهر مفتوحاً

**معايير قبول هذا السيناريو:**

- [ ] أسطر فواتير ومرتجعات بـ CC مختلفة تظهر صحيحة
- [ ] المرتجع يقلل المفتوح (دائن مقابل مدين)
- [ ] دفعة جزئية دون إغلاق إجباري لكل الذمم
- [ ] احترام أيام الاستحقاق (عرض + سياسة توزيع)
- [ ] مقاصة + نقد per CC مع افتراض قابل للتعديل بصلاحية
- [ ] معاينة قيود قبل الترحيل

### سيناريو د — إغلاق مختلط (طرفان + CC)

1. حركات مدينة من عميل على CC1
2. حركات دائنة من مورد على CC2
3. مقاصة جزئية + باقي نقدي
4. قيود متعددة أو قيد مركّب — حسب النمط

### سيناريو هـ — نشر حسب الفرع

1. نمط السند مُقيَّد بفرع «بغداد»
2. الحركات المفتوحة تُفلتر تلقائياً
3. التخصيصات لا تتجاوز فرع السند
4. عند مقاصة مع فرع آخر → قيد بين فروع (إن مُفعّل)

### سيناريو و — عملات مختلفة

1. حركة مفتوحة بعملة USD
2. سند إغلاق بعملة IQD بسعر محفوظ
3. تحقق: التخصيص بعملة الحركة الأصلية أو بعملة أساس موحّدة (قرار تصميم)

## 7) تدفق الشاشة (مستهدف)

```
┌─────────────────────────────────────────────────────────┐
│ نمط السند (إغلاق حركات — مقاصة CC / فرع / طرف)        │
├─────────────────────────────────────────────────────────┤
│ فلاتر: فرع | CC | عميل/مورد/حساب | عملة | مدين/دائن/الكل │
├─────────────────────────────────────────────────────────┤
│ تبويب: حركات مفتوحة (مدين)  │  تبويب: حركات مفتوحة (دائن) │
│  □ قيد #123  CC-A  متبقي 5,000 │  □ قيد #456  CC-B  متبقي 3,000 │
├─────────────────────────────────────────────────────────┤
│ تبويب: مقاصة CC (نقد/لا نقد — افتراض CC، قابل للتعديل بصلاحية) │
│  من CC-A → إلى CC-B  مبلغ: ______  ☑ يشمل نقداً              │
├─────────────────────────────────────────────────────────┤
│ استحقاق: عرض تاريخ الاستحقاق | متأخر | مؤهل للتسديد        │
│ أسطر السند / حساب مقابل / مجموع التخصيصات / تغطية جزئية   │
├─────────────────────────────────────────────────────────┤
│ معاينة القيود الناتجة (قبض/دفع + تسوية CC + تسوية فرع)  │
└─────────────────────────────────────────────────────────┘
```

**ملاحظة:** سطر «استحقاق» يظهر عند ربط الفواتير؛ حتى بدون فواتير مستقلة يُحفظ `due_date` على الحركة المصدر إن وُجد.

## 8) نموذج بيانات مقترح

> للتخطيط — يُنفَّذ على مراحل.

```text
-- توسيع الحركة المفتوحة (منطق أو view)
open_items_view
  journal_line_id, entry_no, account_id, party_id
  cost_center_id, branch_id
  currency_id, original_amount, allocated_amount
  open_amount, open_side             -- debit | credit
  source_invoice_id, source_return_id
  due_date, is_eligible_for_payment, payment_terms_days

cost_centers (توسيع)
  + netting_includes_cash_default boolean

voucher_allocations (توسيع)
  + applied_amount_base
  + target_cost_center_id            -- للتحقق
  + allocation_type                  -- close | netting_cc | netting_branch

voucher_netting_lines (جديد)
  id, voucher_id
  from_cc_id, to_cc_id               -- أو from_branch, to_branch
  amount, currency_id, note

voucher_pattern_closure_settings (جديد — مرتبط بالنمط)
  pattern_id, closure_mode, flags...
```

**خدمة DB مقترحة:** `get_open_items(filters)` — ترجع `open_amount` الصحيح مع خصم التخصيصات المرحّلة فقط.

## 9) قواعد التحقق (إلزامية)

| # | القاعدة |
|---|---------|
| 1 | `sum(allocations) <= voucher_settlement_amount` (حسب تعريف النمط) |
| 2 | كل `applied_amount <= open_amount` للسطر المستهدف |
| 3 | لا تخصيص على سطر غير `posted` |
| 4 | لا تخصيص مكرر يتجاوز الرصيد بعد ترحيل سند آخر (قفل تفاؤلي / trigger) |
| 5 | مقاصة CC: الطرفان مختلفان؛ المبلغ ≤ أصغر رصيد؛ **نقد/لا نقد** حسب CC أو تجاوز بصلاحية |
| 6 | تغطية جزئية: `sum(allocations) <= payment_amount` مسموحة إن `allow_partial_coverage` |
| 7 | استحقاق: سياسة النمط تحدد ما يُعرض كمؤهل للتسديد |
| 8 | فرع: التخصيص ضمن `allowed_branch_ids` للنمط |
| 9 | عملة: سياسة واضحة (عملة الحركة vs عملة السند) — توثيق في النمط |
| 10 | بعد `posted`: التخصيصات غير قابلة للتعديل إلا بعكس السند |

## 10) الترحيل المحاسبي (قيود ناتجة)

سند إغلاق واحد قد يولّد:

| نوع قيد | متى |
|---------|-----|
| قيد قبض/دفع | مبلغ نقدي/بنكي مقابل إغلاق ذمم |
| قيد تسوية CC | مقاصة بين مراكز كلف |
| قيد تسوية فرع | مقاصة بين فروع |
| قيد مركّب | سيناريو مختلط |

**المعاينة قبل الترحيل:** المستخدم يرى كل القيود/الأسطر قبل `posted`.

## 15) قرارات معتمدة وأسئلة مفتوحة

### معتمد

| الموضوع | القرار |
|---------|--------|
| المقاصة تشمل نقداً | **نعم** — افتراض per CC، تعديل بصلاحية `vouchers.closure.override_netting` |

| قبض/دفع عادي | **مسار منفصل** — إغلاق الحركات **لا يستبدله** |
| الفواتير | **يجب بناؤها** بعد توثيق التعريف — ليس قبله |
