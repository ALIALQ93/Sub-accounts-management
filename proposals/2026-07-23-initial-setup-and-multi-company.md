## 0. الوضع الحالي (ملخص)

- المشروع **أحادي الشركة بنيوياً**: جداول singleton (`company_settings` وأخواتها بمفتاح `id=1`).
- لا يوجد `company_id` — الفرع أعلى مستوى تقسيم داخل الشركة الواحدة.
- بذور SQL: فرع `MAIN`، مستودع `WH-MAIN`، 7 حسابات جذر، عملة IQD.
- أول مستخدم في Auth يصبح `admin` عبر `handle_new_user()`.
- لا ويزارد `/setup` ولا عمود `is_setup_complete` قبل هذا التنفيذ.
- RLS مفتوح نسبياً لأي `authenticated` داخل نفس القاعدة — مقبول مع عزل مشروع لكل عميل.

## 1. النموذج المعتمد: نشر منفصل لكل عميل (أ)

كل عميل = مشروع Supabase خاص + نفس كود Next.js بـenv خاص.

**تشغيل عميل جديد:**
1. إنشاء مشروع Supabase.
2. تشغيل `database/setup_all.sql`.
3. تفعيل Email Auth + إنشاء أول مستخدم (Dashboard).
4. نشر التطبيق بـenv العميل.
5. الدخول → `/setup` حتى `is_setup_complete = true`.

تحديثات المنتج تُطبَّق يدوياً على كل نشر (مقبول لعميلين).  
التوثيق: [`docs/deploy-new-client.md`](../docs/deploy-new-client.md).

## 2. مواصفات ويزارد الإعداد (مرحلة B) — تفصيل تنفيذي

### 2.1 آلية الكشف والقفل

| البند | التفاصيل |
|--------|-----------|
| العمود | `company_settings.is_setup_complete boolean not null default false` |
| Patch | `database/patch_setup_complete.sql` — مُضمَّن في `setup_all` |
| ترقية قواعد موجودة | إن وُجد صف في `profiles` → ضبط `is_setup_complete = true` تلقائياً (لا يُجبر العملاء الحاليين على الويزارد) |
| Middleware | إن `false` وطلب محمي → إعادة توجيه `/setup`؛ إن `true` وزيارة `/setup` → `/` |
| استثناءات | `/login`، `/api/*`، و`NEXT_PUBLIC_SKIP_AUTH` |
| عمود ناقص | فشل الاستعلام أو غياب الحقل → اعتبار الإعداد مكتملاً (لا إقفال خاطئ) |
| إعادة فتح | غير مدعومة في v1 (إجراء إداري SQL لاحقاً إن لزم) |

### 2.2 مسار المدير الأول

1. المُشغِّل ينشئ المستخدم من **Supabase Auth → Users** (أو دعوة).
2. المحفّز يمنحه دور `admin` إن كان أول `profiles`.
3. يسجّل الدخول → Middleware يحوّل إلى `/setup`.
4. خطوة «المدير» في الويزارد: تأكيد/تحديث الاسم العربي (والإنجليزي اختياري) — **لا إنشاء Auth جديد من الواجهة في v1**.

> إنشاء مستخدمين لاحقين يبقى عبر `/settings/users` بعد اكتمال الإعداد.

### 2.3 خطوات الواجهة (v1)

| # | الخطوة | الحفظ | ملاحظات |
|---|--------|--------|---------|
| 1 | بيانات الشركة | `company_settings` | اسم عربي إلزامي؛ عملة أساسية؛ بداية سنة مالية؛ هاتف/عنوان اختياري |
| 2 | المدير الأول | `profiles` | عرض البريد للقراءة فقط؛ تحديث الاسم |
| 3 | الفرع والمستودع | `branches` / `warehouses` | تعديل البذور `MAIN` / `WH-MAIN` (رموز وأسماء قابلة للتغيير) |
| 4 | دليل الحسابات | قراءة فقط + قبول | عرض الجذور 1–7؛ لا استيراد قطاعي |
| 5 | إعدادات المخزون | `company_inventory_settings` | جرد + تكلفة (إخفاء FIFO / last_purchase)؛ تحذير القفل بعد أول ترحيل مخزني |
| 6 | القيد الافتتاحي | تخطي | رابط لاحق إلى `/vouchers/opening-entry` — اختياري |
| 7 | إتمام | `is_setup_complete = true` | ملخص + تأكيد → الرئيسية |

### 2.4 واجهة وملفات

| الملف | الدور |
|--------|--------|
| `web/src/app/setup/page.tsx` | صفحة المعالج (بدون شريط تنقل التطبيق) |
| `web/src/modules/setup/*` | حالة الخطوات، واجهات، استدعاءات API |
| `web/src/lib/supabase/middleware.ts` | بوابة `is_setup_complete` |
| `web/src/components/app-shell.tsx` | إخفاء الهيكل على `/setup` مثل `/login` |
| `permission-utils.ts` | السماح بـ`/setup` لأي جلسة مصادَقة |

### 2.5 معايير قبول المرحلة B

- [x] بعد `setup_all` جديد + أول مستخدم: الدخول يفتح `/setup` ولا يُسمح بباقي الشاشات حتى الإتمام.
- [x] إتمام الويزارد يضبط `is_setup_complete` ويمنع العودة لـ`/setup`.
- [x] قواعد فيها `profiles` مسبقاً لا تُجبر على الويزارد بعد تشغيل الـpatch.
- [x] إعدادات المخزون تحذّر من القفل؛ لا تستدعي `lock_company_inventory_foundation` يدوياً من الويزارد.
- [x] FIFO / last_purchase غير ظاهرين.

## 3. النسخ الاحتياطي (مرحلة 1–2 عميل)

| الطبقة | الآن | لاحقاً |
|--------|------|--------|
| نسخ Supabase (يومية / PITR إن Pro) | **إلزامي لكل مشروع عميل** | — |
| `pg_dump` يدوي + حفظ خارجي | **موثّق في دليل النشر** | أتمتة عند ≥3 عملاء |
| اختبار استعادة | يدوي عند أول عميل حي + بعد ترقيات كبيرة | شهري |

لا منتج «نسخ احتياطي داخل التطبيق» في v1.

---

## 4. ما لن نبنيه الآن

- `company_id` / عضوية شركات / تسجيل ذاتي SaaS
- فوترة اشتراكات / بوابة عملاء متعددة على نفس القاعدة
- أتمتة CI لإنشاء مشروع عميل
- قوالب دليل حسابات حسب القطاع داخل الويزارد
- إنشاء المدير الأول من الواجهة بدون Dashboard (لاحقاً إن لزم)
- إعادة فتح الويزارد من واجهة إدارية

## 5. خطة التنفيذ

### المرحلة A — تشغيل عميل (توثيق)

1. [`docs/deploy-new-client.md`](../docs/deploy-new-client.md) — إنشاء Supabase → `setup_all` → env → نشر → نسخ احتياطي → أول مستخدم → `/setup`.
2. قائمة تحقق ترقيات داخل نفس الدليل (أو قسم مضمّن): كل `patch_*.sql` جديد يُعاد على كل نشر عميل؛ لا `setup_all` على بيانات حية.

### المرحلة B — ويزارد الإعداد (منتج)

1. Patch + دمج في `build_setup_all.ps1` / `01_schema.sql`.
2. صفحة `/setup` متعددة الخطوات.
3. Middleware + AppShell + صلاحية المسار.
4. تحذير قفل المخزون في خطوة الإعداد.

### المرحلة C — لاحقاً (عميل ثالث أو ضغط تشغيلي)

1. سكربت شبه آلي لإنشاء مشروع/تطبيق.
2. أتمتة `pg_dump` → تخزين خارجي.
3. إعادة تقييم العزل المشترك فقط إن برّر الحجم التكلفة.
